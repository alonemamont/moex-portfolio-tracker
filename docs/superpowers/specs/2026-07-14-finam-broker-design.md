# Синхронизация остатков с Finam: дизайн

Дата: 2026-07-14

## Контекст и цель

Архитектура синхронизации с брокерами ([[2026-07-13-broker-sync-design.md]])
с самого начала рассчитана на несколько брокеров: `BrokerAdapter`-интерфейс,
реестр (`registry.ts`), diff/apply-логика (`syncDiff.ts`), шифрование токена
(`crypto.ts`), UI-модалки — всё уже broker-agnostic. Сейчас реализован только
Т-Банк (T-Invest API). Цель этой спеки — добавить второй адаптер, **Finam**
(https://api.finam.ru/docs/rest/), той же функциональности и того же скоупа,
что и у Т-Банка: только чтение остатков по акциям и синхронизация в
`sharesOwned`, без торговли и без других типов инструментов.

## Архитектура

Новый модуль `webapp/src/brokers/finam/`, зеркало `tbank/`:

- `client.ts` — сырые REST-вызовы к `https://api.finam.ru`.
- `adapter.ts` — реализация `BrokerAdapter` (`listAccounts`, `fetchHoldings`),
  без изменений интерфейса.

Плюс одна строка в `registry.ts`:

```ts
export const BROKER_REGISTRY: BrokerAdapter[] = [tbankAdapter, finamAdapter];
```

`syncDiff.ts`, `crypto.ts`, `tokenSession.ts`, `runBrokerSync.ts` и все
модалки (`BrokerConnectionsModal`, `AddBrokerConnectionForm`,
`BrokerSyncPreviewModal`) — без изменений, они уже не знают про конкретного
брокера.

## Auth-модель Finam (главное отличие от Т-Банка)

Т-Банк: один статичный bearer-токен, вводится один раз, живёт вечно.

Finam: другая схема — долгоживущий `secret` (API-ключ) обменивается на
короткоживущий JWT через `POST /v1/sessions {secret}`, и уже этот JWT идёт в
`Authorization`-заголовок остальных запросов. Отдельного эндпоинта
"список счетов" нет — список приходит из `POST /v1/sessions/details {token}`
вместе с деталями сессии.

**Решение** (обсуждено с пользователем): `encryptedToken` подключения хранит
именно **secret** — та же AES-GCM/PBKDF2-схема, без изменений формата файла
и без нового поля. `finam/client.ts` сам делает обмен `secret → JWT` перед
каждым реальным запросом (`listAccounts`, `fetchHoldings`), без кэша JWT —
самый простой вариант, минимум движущихся частей; ускорение через кэш JWT в
`tokenSession.ts` отклонено как ненужное усложнение для синхронизации,
которая и так вызывается вручную и нечасто.

Поток синхронизации, экран добавления подключения и разблокировка — без
изменений в UI-слое: поле "токен" в `AddBrokerConnectionForm` в случае Finam
принимает secret, `adapter.listAccounts(secret)` делает обмен внутри себя.

## Поток данных

1. `POST /v1/sessions {secret}` → JWT.
2. `POST /v1/sessions/details {token: JWT}` → список счетов (используется в
   `listAccounts`).
3. `GET /v1/accounts/{account_id}` с `Authorization: Bearer <JWT>` →
   позиции. Подтверждено спайком (2026-07-14): `positions[].symbol`
   (`"SBER@MISX"`, иногда без суффикса биржи) и `positions[].quantity` —
   вложенный объект `{value: string}`, не строка напрямую.
4. Фильтр только акции: `GET /v1/assets/{symbol}?account_id={account_id}`
   (query-параметр обязателен, недокументирован) возвращает
   `{ticker, type}` — держим только `type === "EQUITIES"`.
5. Тикер берётся готовым из поля `ticker` того же ответа — ручная обрезка
   суффикса `@MISX` не нужна, эндпоинт уже отдаёт чистый тикер.

Дальше — тот же `syncDiff`/предпросмотр/подтверждение, что и для Т-Банка, без
изменений.

## Технические риски — закрыты спайком (2026-07-14)

Официальная документация — SPA (Swagger/Redoc), полную JSON-схему ответов
через статический фетч получить не удалось (аналогичная ситуация была и с
Т-Банком). Все три риска ниже проверены вручную (curl) с реальным
read-only secret пользователя (отозван сразу после проверки) в рамках
Task 1 плана реализации:

1. **CORS — открыт.** `api.finam.ru` отдаёт `access-control-allow-origin`,
   эхом отражая `Origin` запроса, и на preflight `OPTIONS`, и на реальный
   `POST`. Браузерный fetch с произвольного origin работает, прокси не
   нужен.
2. **Форма ответа `GET /v1/accounts/{account_id}`.** `positions[].symbol`
   (`"TICKER@MIC"` или без суффикса), `positions[].quantity` — вложенный
   `{value: string}` (не строка напрямую, как предполагалось изначально).
   Тип инструмента в этом ответе не приходит — нужен отдельный резолв через
   `GET /v1/assets/{symbol}?account_id={account_id}` (query-параметр
   обязателен, недокументирован), который отдаёт `{ticker, type}` —
   `type === "EQUITIES"` подтверждено и для акций MOEX, и для иностранных.
3. **Формат `POST /v1/sessions/details`.** Поле `account_ids: string[]`
   подтверждено точно.

Подробности проверки — в `docs/superpowers/plans/2026-07-14-finam-broker-sync.md`,
Task 1.

## Тестирование

Тот же паттерн, что и у `tbank`:

- `finam/client.test.ts`, `finam/adapter.test.ts` — мок `fetch`, без реальных
  запросов к API.
- Кейс в `registry.test.ts` на присутствие `finam` в реестре.
- Ошибка при неверном secret (обмен на JWT падает) обрабатывается тем же
  путём, что и ошибка `listAccounts` у Т-Банка — сообщение в
  `AddBrokerConnectionForm`, без падения приложения.

## Вне скоупа этой спеки

- Другие типы инструментов Finam (облигации, фьючерсы, опционы) — только
  акции, как и у Т-Банка.
- Торговые операции (`OrdersService`) — read-only синхронизация остатков.
- Кэширование JWT между вызовами синхронизации.
- Отчёты (`ReportsService`), корпоративные события, рыночные данные — не
  относятся к синхронизации остатков.

## Автономные решения по ходу дизайна

Пользователь попросил не спрашивать, если рекомендованный вариант почти
наверняка подходит духу проекта, но фиксировать выбор для последующего
разбора. Явно принятые в этой спеке решения без дополнительного вопроса:

- **JWT без кэша** в `tokenSession.ts` — обмен `secret → JWT` делается заново
  при каждом вызове адаптера. Причина: синхронизация и так ручная и нечастая,
  кэш добавил бы состояние ради экономии одного HTTP-запроса за клик.
- **Label и место Finam в UI** — предполагается `label: "Финам"`, добавляется
  в реестр вторым пунктом после Т-Банка, без отдельного текста-подсказки под
  поле токена (в форме нет per-broker подсказок и для Т-Банка). Если по факту
  окажется нужна подсказка про secret vs bearer-токен — добавить точечно.
