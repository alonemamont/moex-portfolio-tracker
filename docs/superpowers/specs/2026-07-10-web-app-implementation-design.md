# Портфель-трекер IMOEX: дизайн реализации веб-приложения

Дата: 2026-07-10

> Продолжение функциональной спеки `2026-07-10-web-app-functional-spec.md`.
> Эта спека закрывает раздел «Вне скоупа» из неё: стек, хранение, файловый
> I/O, хостинг, UI/UX, тестирование.

## 1. Архитектура

- React + TypeScript + Vite, чистый клиент, без бэкенда.
- Прямые fetch-запросы браузера к MOEX ISS. CORS подтверждён вручную —
  ISS отражает `Access-Control-Allow-Origin` под любой origin (проверено
  `curl -H "Origin: https://example.com"` на
  `iss.moex.com/iss/statistics/.../IMOEX.xml`). Прокси не нужен.
- Данные пользователя — единый JSON-файл на диске пользователя, никогда
  не уходит на сервер.
- Хостинг — GitHub Pages, автодеплой через GitHub Actions при пуше в main.
- Состояние приложения — React state/hooks, без Redux/внешних
  стейт-менеджеров (масштаб небольшой, нет нужды).

## 2. Модель данных

Один JSON-файл пользователя:

```json
{
  "version": 1,
  "positions": [
    { "ticker": "SBER", "coefficient": 1.15, "sharesOwned": 100 }
  ],
  "sectors": {
    "SBER": "Финансы"
  },
  "history": [
    {
      "timestamp": "2026-07-10T09:00:00Z",
      "portfolioValue": 123456.78,
      "avgCompliance": 0.02,
      "snapshot": [
        { "ticker": "SBER", "price": 305.5, "weight": 3.1, "status": "in_index" }
      ]
    }
  ]
}
```

- `positions` — только ручные поля (коэффициент, куплено). Live-поля
  (название/цена/вес/лотность/сектор/дивиденд/статус) в файле не
  хранятся, вычисляются заново из ISS + справочника секторов при каждой
  загрузке/обновлении. Файл маленький, live-данные никогда не устаревают.
- `sectors` — **overrides** пользователя поверх встроенного справочника
  (см. раздел 5). По умолчанию пусто, не полная копия справочника.
- `history` — снимок при каждом «Обновить»: агрегаты (стоимость,
  среднее соответствие) + цена/вес/статус по каждой бумаге на тот
  момент (источник графика «цена отдельной бумаги»).
- `version` — задел на будущие миграции формата.
- Схема валидируется при загрузке файла (наличие и типы полей); битый
  файл — явная ошибка, не тихий крах.

## 3. Файловый I/O и первый запуск

- **Загрузка/сохранение** — File System Access API там, где поддержан
  (Chrome/Edge): `showOpenFilePicker()` держит file handle в памяти,
  «Сохранить» пишет в тот же handle без диалога. Fallback (Firefox/
  Safari): `<input type=file>` для загрузки, обычный download-диалог
  для сохранения (каждый раз новый файл в Downloads).
- **Первый запуск / нет файла** — кнопка «Начать с пустого портфеля»:
  тянет текущий состав индекса из ISS, создаёт `positions` (коэф=1.0,
  куплено=0 по каждому тикеру в индексе), `sectors={}`, `history=[]`.
  Первое «Сохранить» — `showSaveFilePicker()` (или download-fallback).
- **Загрузка файла с тикером не из текущего состава индекса** —
  предупреждение в панели ошибок, позиция не отбрасывается.
- **Авто-обновление при загрузке файла** — сразу после выбора файла
  приложение само дёргает ISS и заполняет live-поля (не ждёт ручного
  «Обновить»). Создаёт и пишет снимок в `history`, как обычное
  обновление. Кнопка «Обновить» остаётся для повторных обновлений в
  течение сессии.

## 4. Загрузка рыночных данных и расчёты

- На «Обновить» (и при авто-обновлении после загрузки файла): состав
  индекса — 1 запрос (`limit=100`, обязателен — без него ISS молча
  режет список до 20 строк), цены/лотность — 1 запрос (`limit=100`).
  Дивиденды — параллельно по каждому тикеру, пул ~5-6 одновременных
  запросов (последовательно — секунды vs минуты при 46+ бумагах).
- **Всё или ничего** — только для состава индекса и цен: если хоть один
  из этих двух запросов упал, ошибка в панель, старые данные в
  приложении не трогаются. Единичный проваленный дивиденд-запрос не
  блокирует остальное обновление — дивиденд для этого тикера остаётся
  «неизвестно»/0, как и пустая история выплат.
- Сопоставление по тикеру (регистронезависимо), статусы "в/вне
  индекса", защита ручных полей (коэффициент/куплено) — 1:1 из раздела 3
  функциональной спеки.
- Расчёты (целевое распределение, соответствие индексу, стоимость
  портфеля, доход, среднее соответствие) — чистые функции от текущего
  состояния (`positions` + live-данные), пересчитываются на лету при
  рендере, размер портфеля нигде не захардкожен.

## 5. Встроенный справочник секторов

- `src/data/sectors-default.json` (в репозитории, не в файле
  пользователя) — статичный список тикер→сектор, топ-500 бумаг
  МосБиржи, составляется вручную (ISS официального классификатора не
  даёт — `SECTORID` пусто).
- Итоговый сектор тикера:
  `userFile.sectors[ticker] ?? defaultSectors[ticker] ?? "Другое"`.
- Встроенный список — основной источник. `sectors`-overrides в файле
  пользователя — нишевая фича для редких случаев, когда сектор в
  встроенном списке отсутствует или неверен.

## 6. UI / экраны

Вкладки: **Портфель** / **Графики** / **Сектора**. Общие элементы на
всех вкладках: загрузить/сохранить файл (шапка), боковая панель ошибок
(справа).

### Портфель

Таблица позиций: тикер, название, вес в индексе, цена, лотность,
сектор, дивиденд, статус, коэффициент, куплено (последние два —
редактируемые), целевое распределение, факт. доля, соответствие
индексу, стоимость позиции, доход. Кнопка «Обновить». Итог: общая
стоимость портфеля, среднее соответствие индексу.

### Графики

3 линейных графика (Recharts), источник — `history`:
- Цена бумаги во времени (выбор тикера через dropdown).
- Стоимость портфеля во времени.
- Среднее соответствие индексу во времени.

### Сектора

Круговая/донат-диаграмма распределения стоимости портфеля по секторам
(на основе итогового сектора из раздела 5). Никакой видимой таблицы
справочника — одна кнопка «Изменить сектора», открывающая список
**только тикеров из портфеля пользователя** с возможностью
переопределить сектор (пишет в `sectors`-overrides файла). Полный
встроенный список (топ-500) через UI не просматривается и не
редактируется.

### Боковая панель ошибок

- Справа, фиксированной ширины; длинные сообщения переносятся на
  несколько строк (без горизонтального скролла/обрезки).
- Невидима, когда список ошибок пуст.
- Каждая ошибка — строка с ручным крестиком-закрытием.
- Успешное завершение действия (например, «Обновить» без ошибок)
  авто-очищает ошибки, связанные с этим действием.

## 7. Обработка ошибок

Переносится 1:1 из функциональной спеки (раздел 8):
- ISS недоступен → ошибка в панель, старые данные не трогать.
- Тикер делистнут/не торгуется → цена не обновляется (остаётся
  последнее известное значение), статус = "вне индекса".
- Пустая история дивидендов → 0, не ошибка.
- Тикер из загруженного файла вне текущего состава индекса →
  предупреждение в панель, не отбрасывается.

Плюс специфика веб-реализации:
- Ошибка парсинга/валидации JSON-файла при загрузке — явное сообщение
  с указанием проблемного поля, приложение не падает молча.
- FSA API недоступен (Firefox/Safari) или пользователь отменил
  диалог — не ошибка, тихий fallback/no-op.

## 8. Тестирование

- Unit-тесты (Vitest) на чистые функции: расчёты (целевое
  распределение, соответствие, стоимость, доход, среднее соответствие
  с делением на 0/пустой список), сопоставление по тикеру (life-cycle
  правила раздела 4), merge встроенного справочника секторов с
  overrides пользователя.
- ISS-клиент — тесты на моках fetch (XML-парсинг, пагинация
  `limit=100`, all-or-nothing при ошибке состава/цен).
- Без e2e на первом этапе — ручная проверка в браузере перед мержем
  (CI/деплой вне скоупа этой спеки, кроме автодеплоя GitHub Pages).

## Вне скоупа этой спеки

- Наполнение `sectors-default.json` реальными данными (топ-500 тикеров
  и их секторов) — отдельная задача по сбору данных, не архитектурная.
- Миграции формата файла при будущих изменениях схемы (`version`
  зарезервировано, сама логика миграции не спроектирована).
- Мультиустройства/синхронизация, аккаунты — явно вне скоупа
  (функциональная спека, модель «один файл, без логина»).
