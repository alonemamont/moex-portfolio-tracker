# Синхронизация остатков с брокерами: дизайн

Дата: 2026-07-13

## Контекст и цель

Портфель пользователя часто разбит между несколькими брокерами (Т-Банк,
БКС, Сбер...). Сейчас `sharesOwned` — чисто ручное поле, которое нужно
поддерживать вручную после каждой сделки. Цель — забирать фактические
остатки по акциям напрямую из брокерских API, по одному или нескольким
подключённым брокерам одновременно, без сервера и без ручного экспорта
отчётов.

Первая реализация — адаптер под **T-Invest API** (Т-Банк). Архитектура
с самого начала рассчитана на несколько брокеров одновременно: портфель
может собираться из T-Invest + БКС + вручную введённого остатка (счёт,
для которого адаптера ещё нет).

## Архитектура

Клиентское приложение остаётся полностью статическим (GitHub Pages, без
бэкенда) — прямые `fetch`-запросы из браузера к REST-прокси брокерских
API с Bearer-токеном пользователя.

Новый модуль `webapp/src/brokers/`:

- `types.ts` — общий интерфейс адаптера:
  ```ts
  interface BrokerAdapter {
    id: string;        // "tbank"
    label: string;      // "Т-Банк"
    listAccounts(token: string): Promise<{ id: string; name: string }[]>;
    fetchHoldings(token: string, accountId: string): Promise<{ ticker: string; shares: number }[]>;
  }
  ```
- `tbank/` — реализация адаптера под T-Invest API (`client.ts`,
  `adapter.ts`).
- `registry.ts` — список доступных адаптеров (сейчас — только `tbank`).
  Добавление нового брокера в будущем = новая папка + запись в реестре,
  без изменений в остальном приложении.
- `crypto.ts` — шифрование/расшифровка токена (см. ниже).
- `syncDiff.ts` — чистая функция построения предпросмотра изменений.

Только T-Invest реализуется сейчас. БКС/Сбер и другие — вне скоупа этой
спеки, архитектура их не блокирует.

## Модель данных

### Множественные источники количества акций

`sharesOwned` перестаёт быть единственным числом — становится суммой
источников. Обратная совместимость: существующее поле `sharesOwned` в
файле сохраняет смысл "ручной остаток/остаток без брокера" — старые
файлы грузятся без миграции, `brokerHoldings` по умолчанию пуст, итог
равен старому значению.

`file/schema.ts`, добавление к позиции:

```ts
positions[].sharesOwned: number         // ручной остаток (как сейчас)
positions[].brokerHoldings?: {
  connectionId: string;                  // ссылка на brokerConnections[].id
  shares: number;
  syncedAt: string;                      // ISO-дата последней синхронизации
}[]
```

`CalculatedPosition.sharesOwned` (вычисляемый тип, не файловый) меняет
смысл на **итог**: `file.sharesOwned + sum(brokerHoldings.shares)`.
Все существующие потребители (`buildCalculatedPositions`, таблица,
карточка, расчёты) продолжают читать `CalculatedPosition.sharesOwned`
как единое число — их код не меняется, меняется только откуда это число
берётся.

Редактирование ячейки "Куплено" в таблице/карточке остаётся как есть и
правит `file.sharesOwned` (ручную часть) — тот же callback
`onChangeSharesOwned`, то же поведение, если у позиции нет ни одного
брокерского источника.

Когда у позиции есть хотя бы один брокерский источник — рядом с итогом
показывается разбивка по источникам (тултип, тот же паттерн `th-hint`,
что уже используется в таблице): `Т-Банк: 10, Вручную: 2 = 12`.

### Брокерские подключения

Новый раздел файла:

```ts
brokerConnections?: {
  id: string;             // uuid, генерируется при добавлении
  brokerId: string;       // "tbank" — ключ в registry.ts
  accountId: string;      // id счёта у брокера
  label: string;          // "Т-Банк — брокерский счёт"
  encryptedToken: {
    ciphertext: string;   // base64
    iv: string;           // base64
    salt: string;         // base64
  };
}[]
```

Один брокер может быть подключён несколько раз с разными `accountId`
(например, брокерский счёт и ИИС в Т-Банке отдельно) — каждое
подключение получает собственный `connectionId`, на который ссылаются
`brokerHoldings`, так что повторная синхронизация одного подключения не
задевает вклад остальных.

## Хранение и шифрование токена

Токен брокера кладётся **в `portfolio.json`**, зашифрованным —
пользователь просил перенос между устройствами вместе с файлом портфеля,
а не привязку к одному браузеру.

- Шифр: AES-GCM-256 через `SubtleCrypto` (WebCrypto), ключ — PBKDF2 от
  пользовательской пароль-фразы (соль на подключение, много итераций).
  Никакой самописной криптографии.
- **Пароль-фраза нигде не хранится** — ни в файле, ни в `localStorage`.
  Вводится пользователем при добавлении подключения (шифрование) и при
  разблокировке после загрузки файла (расшифровка).
- Расшифрованный токен на время сессии кэшируется в `sessionStorage`
  (переживает reload вкладки, исчезает при закрытии вкладки/перезапуске
  браузера) — один ввод пароль-фразы на сессию, а не на каждую перезагрузку
  и не на каждый запуск при том же файле в той же вкладке.
- Если файл загружен на новом устройстве/в новой вкладке — подключения
  показываются как заблокированные (🔒), с кнопкой "Разблокировать",
  пока не введена пароль-фраза. Неверная пароль-фраза — ошибка расшифровки,
  чёткое сообщение "неверный пароль", без падения приложения.
- **Явное предупреждение в UI** при добавлении первого подключения:
  выгружая/передавая `portfolio.json` дальше, пользователь передаёт и
  зашифрованные токены — безопасность целиком зависит от стойкости
  пароль-фразы.

## Поток синхронизации

Точка входа — новый пункт в `Header` ("Брокеры"), в т.ч. в мобильном
`⋮`-меню: открывает `BrokerConnectionsModal`.

**Добавление подключения:**
1. Выбор брокера из `registry.ts` (сейчас — только Т-Банк).
2. Ввод токена → `listAccounts(token)` → выбор счёта из списка.
3. Ввод пароль-фразы → шифрование токена → подключение добавляется в
   `file.brokerConnections`.

**Синхронизация (по кнопке на конкретном подключении):**
1. Расшифровать токен (если ещё не разблокирован в сессии — запросить
   пароль-фразу).
2. `fetchHoldings(token, accountId)` → список `{ ticker, shares }`.
3. Сопоставление по тикеру с текущими позициями — те же правила, что и
   при обновлении рыночных данных (регистронезависимо, по тикеру, не по
   порядку).
   - Тикер уже есть в портфеле → предпросмотр "было → станет" для
     вклада именно этого подключения.
   - Тикер новый, но резолвится на бирже (есть в текущем составе
     индекса/на MOEX) → предпросмотр "новая позиция", коэффициент 1.0.
   - Тикер не резолвится (не акция/делистинг/не найден) → пропуск с
     явным предупреждением в предпросмотре, не молчаливое отбрасывание.
   - Тикер, который раньше был синхронизирован с этого подключения, но
     отсутствует в новом ответе → вклад этого подключения обнуляется
     (авторитетный источник по своему счёту — "всё или ничего" в рамках
     одного подключения, другие подключения и ручной остаток не
     трогаются).
4. Пользователь видит таблицу изменений и подтверждает — только после
   этого правки применяются к `file`. Ничего не пишется без подтверждения.

**Ошибки сети/CORS:** если запрос к брокерскому API падает (в т.ч. по
CORS — не проверено эмпирически на момент дизайна, брокерские API часто
не открывают CORS для браузеров), показывается понятная ошибка
("не удалось подключиться, возможно ограничение брокера"), синхронизация
для этого подключения недоступна, остальное приложение не страдает.

## Открытые технические риски (спайк перед кодированием остального)

1. **CORS.** Не проверено, отдаёт ли `invest-public-api.tbank.ru`
   заголовки, разрешающие браузерный fetch с произвольного origin.
   Первый шаг реализации — ручная проверка через DevTools/минимальный
   fetch. Если CORS блокирует — фича для Т-Банка нереализуема без
   прокси-сервера, что нарушает архитектурный принцип "без бэкенда";
   в этом случае решение по этой ветке принимается отдельно (не
   блокирует остальную архитектуру — реестр адаптеров всё равно полезен
   для будущих брокеров).
2. **Форма ответа `GetPortfolio`.** Точная JSON-схема не подтверждена
   (SPA-документация брокера не отдаёт схему статическому фетчу).
   Неизвестно, приходит ли `ticker` напрямую в `positions[]` или только
   `figi`/`instrumentUid`, что потребует дополнительного вызова
   `InstrumentsService.GetInstrumentBy` для резолва тикера. Проверяется
   тем же ручным спайком.

## Тестирование

- Чистые функции — юнит-тесты по конвенции репозитория:
  `syncDiff.ts` (построение предпросмотра, обнуление отсутствующих
  тикеров, склейка нескольких подключений), `crypto.ts` (шифрование —
  расшифровка round-trip, неверный пароль → ошибка), маппинг ответа
  `tbank`-адаптера в общий формат.
  `buildCalculatedPositions`/производные — тест на суммирование
  `sharesOwned + brokerHoldings`.
- `client.ts` (сетевые вызовы) — мок `fetch`, без реальных запросов к
  API в тестах.
- Модалки (`BrokerConnectionsModal`, предпросмотр диффа) — ручная QA по
  текущей конвенции репозитория (компонентных тестов на модалки в
  проекте пока нет); если к моменту реализации в проект уже попадёт
  `@testing-library/react` (см. `2026-07-13-mobile-usability-design.md`),
  добавить и компонентные тесты.

## Вне скоупа этой спеки

- Адаптеры под БКС, Сбер и другие брокеры — только архитектура под них,
  без реализации.
- Импорт отчёта брокера (CSV/Excel) как альтернативный путь — не нужен,
  раз есть прямой API-путь.
- Синхронизация чего-либо, кроме `sharesOwned` (например, истории
  сделок, цены приобретения, налоговой информации).
- Автоматическая периодическая синхронизация в фоне (нет фонового
  сервера — только по явному действию пользователя, как и обновление
  рыночных данных).
